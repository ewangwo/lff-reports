## 一、三形式报告

**定义：** 三类交付物，不是三种文字口径。
- ① 自用MD报告 — 可含LFF私人持仓、约束、行动含义
- ② 微信图片/信息图 — 高密度、DeepSeek风格，公共/转发安全，不包含私人持仓
- ③ HTML网页版 — deploy到 ewangwo.github.io/lff-reports/，公共安全

**Source-Completeness 标准（9要素）：**
1. 源内容身份（标题/作者/日期/来源）
2. 核心概念层
3. 公式/机制/模型
4. Proof point 表格
5. 路线图
6. 系统架构
7. 挑战矩阵
8. 数据全景（关键数字集中展示）
9. 深层断言 + LFF判断

**时间预算：**
| 节点 | 目标 |
|------|------|
| T+5min | 给出可读 Markdown / source digest |
| T+8min | 汇报已完成文件、剩余步骤、是否需要加速 |
| T+10min | Markdown+微信图+网页 全部完成或告知用户 |

**GitHub Pages：**
- 仓库：`ewangwo/lff-reports`，Pages：`https://ewangwo.github.io/lff-reports/`
- HTML 放 `reports/` 目录
- Codex 更新 `index.html`，Hermes 更新 `hermes.html`
- 推送上不去时：聊天直接展示内容，注明"待补推"
- push命令：`ssh://git@ssh.github.com:443/ewangwo/lff-reports.git`

## 二、五源采集

**当前规则（Air Codex 执行，Pro 不直接操作）：**
- 入口：Google Chrome existing-tabs-only，不使用 Safari
- 前置：`python3 scripts/five_sources_daily.py --preflight`
- 通过条件：`all_selected_sources_matched: true` + `js_apple_events.ok: true` + 无临时Chrome冲突
- 采集：预检成功后对每个匹配标签刷新一次 → 等待reload → 抓取最新内容
- 失败：标注失败原因，不用旧缓存冒充
- 高风险页面（KYC/验证码/二次验证）：滚动读取，不主动刷新，标注"未刷新"

**输出标准：**
1. 抓取状态表（来源、时间、字符数、是否刷新后实时）
2. 每源关键内容（3-6条核心+关键数字）
3. 跨源综合主题
4. LFF结论/行动含义
5. 正文直接展示，路径仅作附录

## 三、高价值问题门禁

**触发域：** 资金/投资/券商/账户、工作/职业/现金流、健康/体重/血糖、家庭/伴侣、用户纠错

**流程：**
```
Step 1: 判断领域 → 命中高价值域
Step 2: 8源最低检索
  - MEMORY.md / USER.md
  - 👤 用户信息.md / 📈 核心持仓深度分析.md
  - 🧠 LFF_行为与投资准则.md / 💡 投资理解与分析框架.md
  - 📥 缓存/ + (如提到Hermes) sessions/ + cron/output/
Step 3: 生成内部 context card（三行）
  Known facts: 回答用到的已知事实
  Unknown/stale: 哪些找不到或过期
  Implication: 这些事实如何改变结论
Step 4: Anti-generic 4问
  - 对任何陌生人成立？
  - 在问已有答案的问题？
  - 映射到LFF真实账户/持仓/健康？
  - 标注快照日期和数据边界？
Step 5: 输出 — 删除泛化建议，保留个人化内容
```

**最小承诺：** 不跳过高价值识别 → 不未检索就给重大建议 → 不把已知当未知 → 不输出泛化建议 → 找不到就明确说不知道

## 四、输出质量标准

每次输出前自查8项：
- [ ] 结论在开头1-2句内给出？
- [ ] 关键数据标注来源和日期？
- [ ] 有明确的行动建议或风险提示？
- [ ] 区分事实（据XX数据）和观点（我的判断）？
- [ ] 格式结构化（表格/列表 > 散文）？
- [ ] **正文直接展示，非仅文件路径？**
- [ ] **Anti-generic 4问已通过？（高价值问题）**
- [ ] **超10秒操作有进度更新？**

## 五、SLA执行纪律

| 节点 | 必须做到 |
|------|----------|
| T+0 | 告知用户"正在做" |
| T+20s | 说明任务理解和第一步动作 |
| T+60-120s | 给出初步源摘要/文件定位 |
| **T+5min** | **必须输出最小可用结果 ← 核心红线** |
| T+8-10min | 完成主要产物或汇报剩余blocker |
| T+10min+ | 必须拆阶段，不能继续沉默 |

**300秒看门口：** 无meaningful progress时中断业务流→停止循环→切换路线→输出中断报告（环节/卡点/已尝试/替代路线）

## 六、8条绝不能犯的错

1. 五源采集时新开Chrome、跳转已有标签、刷新登录页面
2. 使用Safari做五源（除非用户明确要求）
3. 报告/图片/总结只给文件路径，不展示正文或图片
4. 文件查找只扫一个目录
5. 投资/研究回答不映射LFF持仓和约束
6. 遇到失败只说"无法获取"，不列尝试记录和替代方案
7. 长时间操作静默等待
8. 面向同事传播的图片加入LFF个人持仓或个人建议

## 九、审计流程（7步）

```
1. Load agreement — 读同步协议/当前规则/MEMORY/服务协议
2. Age scan — frontmatter updated vs mtime，超14天列stale
3. Owner boundary — 用户事实不混入Agent协议
4. Conflict scan — 相反规则列出并统一
5. Link sample — 抽样wikilink，修P0入口链接
6. Overload — 缓存>200文件提示索引缺口
7. Remediation — P0直接修，P1标记，P2记录
```

Stale阈值：标题含"当前/最新/高频/索引/快照"的文件，updated超14天必须标记stale。
